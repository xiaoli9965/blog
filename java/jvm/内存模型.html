      # title: JVM&&JMM(8版本)

      JVM相关知识

      • JVM 不仅仅只能运行 Java 程序，还能运行 Scala，Groovy，kotlin 等多种语言（目前约 10 种）,JVM 官方有一套官方规范，只要符合规范的文件都可以在 JVM 中运行
      • 目前主流虚拟机叫 🔥HotSpot（Oracle的），还有 N 多其他的公司的虚拟机,比如 JRockit VM （SUN 公司）,OpenJDK(开源的)， IBM 的 J9 VM 等等；
      • JDK8 的方法区改名为 MetaSpace（元空间），直接存放在物理内存上

      Hotspot JVM 后台运行的系统线程主要有下面几个：

      所有线程共享:

      • Method Area

      • Head堆

      • 线程隔离数据区

        • 程序计数器:
          • BPC (bytecode pointer),在程序执行时标记程序执行到哪里了
        • 虚拟机栈 VM Stack: 每个方法执行都会创建一个栈帧, 用于存储局部变量表、操作数栈、动态链接、方法出口等信息
          • 局部变量表: 保存栈执行过程中的常量
          • 操作数栈: 先进后出的栈
          • 指向常量池的指针
        • 本地方法栈( Nvtive Method Stack): 本地方法栈则为 Native 方法

      • 线程共享区域【JAVA 堆、方法区】、直接内存

        • 堆: 运行时数据区
        • 方法区: metaspace 运行时常量池

      # 1. 对象内存存储

      • Markword： 存放锁状态，对象分代，hashcod 等,不压缩占 8 个,压缩占 4 个
      • Class point： 存放当前对象类型的引用,所有的 Class 类型都会被存放点 MetaSpace 中
      • Instance Data： 每个变量指针占 8 个字节,开启压缩占 4 个
      • Panding： 如果前面几项加起来字节数不能被 8 整除，就会使用这个区域行补齐..如果满足被 8 整除这个区域则没有

      # 2. 对象压缩

      JVM 中默认开启了对象压缩,在内存超过 32G 时对象压缩自动失效; 如果开启压缩了:Markword：占 4 个字节,Instance Data：每个占 4 字节;

      所以原版在 32G 内存中运行够用,换到 64 的机器后发现内存没多出来多少就是因为对象压缩关闭了 可能原本在 32G 环境中占 30G 内存,进行内存扩容后 30G 的数据直接变成了占用 45G

      # 2.1. 栈上分配

      JDK8 默认开启栈上分配,栈上分配是特别爽的事情,因为方法执行完栈就自动弹出了(内存释放),不用考虑垃圾回收的问题

      在分代模型中,当 new 对象的时候首先在栈上分配(栈上能分配的情况下),否则就判断对象大小,如果太大直接放入老年代,如果不大就判断能否在 TLAB 存放,不能则存入栈中

      # 2.2. TLAB (Tread Local Allocation Buffer)

      本地线程缓冲区

      JVM 会为每个线程在堆中开辟一小块空间用于存放这个线程的对象;

      为什么要有 TLAB? 一个 JVM 中会运行 N 个线程,如果同时创建对象的话会存在内存区域竞争问题.
      假如一个线程要要创建对象在 0x01 区域,另一个线程也要创建对象在 0x01 区域 这个时候就存在锁竞争问题(CAS 锁)会导致 CPU 资源浪费,所以有了 TLAB 避免了一大部分的竞争问题.

      Thread Local Allocation Buffers，线程本地分配缓存。JVM使用了TLAB这种线程专属的区间来避免多线程冲突（无锁方式），提高对象分配效率。TLAB本身占用了Eden空间，即JVM会为每一个线程都分配一块TLAB空间
