    # 1. Monitor

    内容来自 B站 (opens new window)

    同步代码块 汇编代码

    对应的汇编

    可以看到在同步代码块前后会有一个 monitorenter 和 monitorexit

    # 1.1. monitorenter 进入锁

    • 每个对象都会有一个监视器(monitor)关联,监视器被占用时锁住,其他线程无法获取该monitor

    monitor的两个属性

    • owner: 记录拥有锁的线程
    • resursions: 记录锁的重入次数

    小结

    1.synchronized的锁对象会关联一个monitor,这个monitor不是我们主动创建的. 2.JVM的线程执行到这个同步代码块时,发现锁对象monitor没有创建就会创建一个. 3.当线程拥有monitor时其他线程只能等待

    # 1.2. monitorexit 结束锁

    每当执行完一个同步代码块时 monitor中的Resursions就会进行减一, 当resursions为0时就会清除到Owner变量释放掉这个monitor

    # 1.3. 汇编码出现两次monitorexit的原因

    第二次出现monitorexit是为了防止同步代码块中的代码出现异常,,当出现异常时就会进入第二个monitorexit进行释放锁

    # 1.4. 同步方法

    • 同步方法的汇编代码并不会出现monitorenter 和 monitorexit
    • 同步方法中有以ACC-SYNCHRONIZED标记来进行隐式调用monitor

    # 2. Monitor 原理

    Monitor对象

    • _recursions: 重入次数
    • _objiect: 记录Java对象的引用
    • _WaitSet: 处于wait等待的Java线程
    • _cxq: 获取锁失败的线程
    • _entryList: 获取这个锁处于blocked状态的线程

    Monitor示意图

    # 2.1. Monitor 锁竞争

      1. 通过CAS操作吧monitor的Owner字段设置成当前线程
      1. 如果monitor的Owner为当前线程(说明是重入锁),则只需Recusions++ 来记录重入次数
      1. 如果当前线程第一次进如该monitor,则设置Recusions字段为1 (成功获取锁)
      1. 如果获取锁失败,则存入_cxq队列等待锁释放

    # 3. synchronized 为什么是重量级锁

    因为monitor的函数调用中会涉及到 Atomic::cmpxchg_ptr , Atomic::inc_prt等内核函数. 锁竞争时会用到park,unpack等函数,这个时候就会就会调用系统内核进行态切换

    所以 synchronized是重量级锁

    # 3.1. 锁升级

    在JDK1.5之前syc是一个重量级锁,在1.5-1.6的时候对synchronized作出了重要改进(因为JDK源码中大量用到了sync要想提升效率必须做优化)

    锁升级过程(JDK1.6以后)

    无锁 > 偏向锁 > 轻量级锁(自旋锁) > 重量级锁

    # 3.2. Java对象布局

    Java对象布局:

    • 对象头
    • 实例数据
    • 对齐填充

    • Markword： 存放锁状态，对象分代，hashcod 等,不压缩占 8 个,压缩占 4 个
    • Class point： 存放当前对象类型的引用,所有的 Class 类型都会被存放点 MetaSpace 中
    • Instance Data： 每个变量指针占 8 个字节,开启压缩占 4 个
    • Panding： 如果前面几项加起来字节数不能被 8 整除，就会使用这个区域行补齐..如果满足被 8 整除这个区域则没有

    # 3.3. 偏向锁

    标志位:01

    hotSpot作者研究发现,大部分情况锁时不存在竞争的,而是由同一个线程多出获得锁,释放锁,所以对获取锁做了个优化(偏向锁)

    偏向锁会在锁对象的对象头中以52个字节存储锁ID
    2个字节记录时间
    1个字节存储锁状态(无锁,偏向锁,自旋锁,重锁)

    • 1.偏向锁仅适用于无线程竞争的状态(一个线程多次执行同步代码块)
    • 2.如果有多个线程获取锁会立马升级为自旋锁

    # 3.4. 轻量级锁(自旋锁)

    标志位:00

    在多线程交替执行同步代码块的情况下,就行的锁优化(没有锁竞争).
    如果有多个线程同时竞争锁就会升级到重量级锁

    # 4. 平时写代码对synchronized的优化

    减少synchronized范围

    • 在同步代码块中尽量少写点代码,这样执行就会比较快,在时间上可以避免一些不必要的锁竞争导致锁升级

    降低锁粒度

    • 将一个锁拆分成多个锁,提高并发.. 例:hashTable
