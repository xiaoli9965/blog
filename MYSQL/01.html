        # 4. InnoDB

        # 4.1. 从磁盘加载数据

        • 局部性原理
        • 页16kb, (4KB的倍数)

        # 4.1.1. 双写缓冲

        页的备份, 防止页写一半写失败了

        操作系统的页是 4K一个, innodb的是16k, 那么就需要写4次.. 双写缓存就是防止页部分失败的机制

        # 4.2. InnoDB buffer pool

        将内存pool的数据刷入磁盘就叫刷脏 为了防止内存数据没刷到磁盘中,提高可靠性,避免数据丢失从来产生了: redo log

        在liunx系统中,默认的buffpoll是128m, 最大能用到服务器内存的80%以上

        为什么要引入 buffer pool 而不直接写入磁盘? 因为顺序IO 速度快, 顺序IO降低了磁盘寻址的销毁

        内存空间写满了会怎样? LRU缓存, 缓存并不是直接指向具体数据,而是指向数据对应的数据页的指针 分为冷数据和热数据区,淘汰会直接淘汰冷数据区

        • 冷数据区: 3/8
        • 热数据区: 5/8

        任何数据进来的时候都会到达冷数据区

        # 4.3. redo log

        redo log是顺序刷盘的, redo log也有缓冲区 叫log buffer

        默认大小 48m, 当48m写满后会进行 buffer pool刷盘

        存储路径: 默认在安装目录下, 默认有2个文件 名字叫 ib_logfile0, ib_logfile1

        # 4.3.1. 刷盘时机

        刷盘时机要根据性能和可靠性取个平衡点, 默认值commit之后就会立即刷入磁盘

        • 性能越好那么刷盘间隔就要越长
        • 可靠性越高就要输盘评率越高

        # 4.4. undo log (事务)

        记录事务发生之前的数据状态,发生异常是回滚,保证原子性

        • 一个数据多条事务,会形成undolog链, 和 ROW_RELL_PNT有关

        # 4.5. binlog

        这个是Server层的日志, 以事件的形式几率了所有的DDL 和 DML语句, 可以用来做主从复制和数据恢复

        # 4.6. 数据更新过程(简化)

        1. 事务开始, 从内存(buffer pool)或磁盘(data file)取到包含这条数据的数据页,返回给Server的执行器;
        2. Server的执行器修改数据页的这一行数据值为 'alexli';
        3. 记录neme='alexli'到undo log;
        4. 记录name='alexli'到redo log;
        5. 调用存储引擎接口,吉利数据页到buffer pool;
        6. 提交事务;

        # 4.7. change buffer (Insert Buffer)

        change buffer是buffer pool的一部分, 默认占用buffer pool四分之一的空间(25%) 如果这个数据页不是唯一索引,非唯一性的数据. 这种情况可以先把修改的数据存到change buffer中, 从来大大提升 DML,DDL的执行速度.

        # 4.8. 自适应Hash索引

        在建表时指定 using hash,这个是innodb为热点数据自动添加的hash索引,为了提升热点数据访问速度

        # 5. 索引

        索引底层用的 B+Tree, 就是 B-TREE+链表的一个结构

        为什么不用二叉树?

        • 当数据量大了之后,树的深度会很深要查找很多次
        • 在某种极端情况下,二叉树会退化为线性结构类似于链表

        # 5.1. innoDB 影藏的字段

        innodb默认会为表简历三个影藏字段

        • **DB_ROW_ID: **6字节 行标识 自增
        • DB_TRX_ID: 6字节 事务ID 自增
        • DB_ROLL_PTR: 7字节 回滚指针,删除版本号

        # 5.2. B + Tree

        优势:

        • 扫库,扫表能力更强
        • 磁盘读写能力更强
        • 排序能力更强
        • 效率更加稳定
        • 一般到3层的时候就能存储千万级别的数据了,

        # 5.3. InnoDB二级索引

        非聚集索引的索引列全部都是二级索引

        • 二级索引的值记录这主键id, 查询的时候会回表查局促索引

        为什么二级索引不直接记录整条记录的磁盘索引? 因为数据表会一直不断新增变化,变化后主键需要重新排序维护磁盘位置也会变动. 如果二级索引直接记录磁盘位置,那么在DML的时候还需要额外的去维护二级索引.

        # 5.4. 自动建立索引的规则?

        • 如果没有建立索引, innodb会找到第一个不包含空值的字段建立唯一索引;
        • 没有主键索引,但是有普通索引, 那么第一个普通索引就是聚簇索引;
        • 如果没有包含非空值的字段, 会生成一个隐藏的列 row id(自增,6字节) 作为主键索引

        # 5.5. 离散度

        • 不要在离散度低的字段建立索引(比如性别字段), 这种重复率很高的字段建立索引是没有意义的, 起不到提速的作用

        # 5.6. 最左匹配原则

        联合索引会根据建立时 字段顺序进行排列的

        • 查询时 想要走联合索引,必须要有 第一个字段 作为条件,否则失效
        • 查询时 多个字段 顺序可以随意填写, server层有优化器会进行处理

        # 5.7. 索引条件下推

        innodb自动开启,自动优化的. 用户无法控制;

        本来数据的过滤是在server层完成的, 在server要做的操作先在存储引擎层完成.这就叫索引下推

        打个比方:

        • 联合索引的(a,b) , 比如 a='x' like b='%x' 的查询, 正常情况是 存储引擎查到 a=x的数据然后返回给server层, 然后server再进行like b='%x'过滤
          • 如果根据a='x'查询出来10W条数据,但是 满足like b='%x'的数据只有一条,那么就极大浪费了server的计算能力
        • 这个时候存储引擎层会进行索引下推,在引擎层完成数据的过滤
          • 存储引擎先根据a等值查询索引,然后拿到a的b字段数据,再去联合查询.. 这样就是两次索引查询然后返回server层

        # 5.8. 建立索引规则

        • where \ order排序 \ join \ group by 的字段上建立索引
        • 索引个数不要建立太多 (浪费磁盘,修改数据还需要维护索引)
        • 区分度低的字段, 例如 性别, 不要建立索引
        • 复合索引吧离散度高的字段放到前面
        • 频繁更新的值,不要建立主键或者索引
        • 不建议使用无序的值建立主键索引 (身份,UUID) , 二级索引无所谓
        • 过长字段,建立前缀索引, 前缀索引可以指定截取多少长度作为前缀索引.

        # 5.9. 什么情况用不到索引

        • 索引列上使用函数 (reolace\ substr\concat\ sum\ count\avg), 表达式
        • 字符串不加引号,出现隐式转换
        • like条件前面带 %

        # 6. 事务

        # 6.1. ACID

        • 原子性 (undo log)
        • 隔离性 (锁 + MVCC)
        • 持久性 (redo log , double wirt buffer)
        • 一致性

        # 6.2. 脏读\幻读\不可重复读

        脏读: 读到其他事务未提交的数据(读到了内存中的脏页)

        幻读: 2次相同的查询操作. 第一次读1条,第二次读变成了2条

        不可重复读: 两个事务,事务1前后两次读取的数据值不一样, 因为读到了事务2已提交的数据

        # 6.3. 四种隔离级别 (SQL 92标准)

        • Read Uncommitted (RU 读未提交):
          • 未解决任何并发问题 ( 生产环境不会使用⭐️)
        • Read Committed (RC 读已提交): 解决脏读
          • MVCC + 行锁 + 间隙锁 + 临键锁
        • Repeatable Read (RR 可重复读): 解决不可重复读
          • MVCC + 行锁
        • Serializable (串行化): 最高隔离级别,解决所有并发问题
          • ( 生产环境不会使用⭐️) 所有的语句都会加锁,读会阻塞增删改

        InnoDB 可能出现幻读 下

        # 6.4. RR RC 怎么选择

        推荐RC, 如果你对锁特性很了解可以使用RR

        1. RR的间隙锁导致锁定范围扩大, 导致并发降低
        2. 条件列未使用到所有, RR锁表,RC锁行

        # 6.5. MVCC

        Mutli Version Cncurrency Control 多版本并发控制

        效果: 建立一个快照, 同一个事务无论查多少次都是相同的数据

        一个事务能看到的数据版本:

        • 第一次查询之前已提交的事务修改
        • 本事务的修改
        • RC的 "半一致性 Semi Consistent " 可以增加update的并发性能;

        一个事务不能看见的数据版本:

        • 在本事务第一次查询之后创建的事务(事务ID比我大)
        • 活跃的(未提交的)事务

        # 6.5.1. MVCC如何实现

        Read View 一致性视图 每一次事务开启的第一次查询之后,会建立一个 read view.

        Read View包含一下几个内容:

        • m_ids{} : 当前系统活跃的事务ids
        • min_trx_id: m_ids的最小值
        • max_trx_id : 系统分配给下一个事务的id
        • creator_trx_id : 生成read view的事务id

        # 6.5.2. RV建立时机

        • RR: 事务中第一次查询就建立了快照,以后永远不会变
        • RC: 查一次建立一次MVCC, 所有会读到脏数据

        # 6.5.3. Read view的判断规则

        • 从数据的最早版本开始判断 (undo log)
        • 从数据版本 trx_id = creator_trx_id, 本事务修改
          • 本事务修改, 可以访问
        • 数据版本 trx_id<min_trx_id (事务提交最小规则)
          • 说明这个版本在生成RV已经提交, 可以访问
        • 数据版本的 trx_id > max_trx_id (下一个事务id)
          • 这个版本是生成RV之后建立的, 不可以访问
        • 数据版本的trx_id 在 min_trx_id和max_trx_id之间
          • 看看是否在m_ids中,如果在不可以. 如果不在就可以
        • 如果当前版本不可见,就找 undo log链中的下一个版本;

        # 6.6. 锁

        • 共享锁 (行锁): Shared Locks

        • 排他所 (行锁): Exclusive Locks

        • 意向共享锁 (表锁): Intention Shared Locks

        • 意向排它锁 (表锁): Intention Exclusive Locks

        • S锁: 共享锁

        • X锁: 排它锁

        ​

        一个事物所加的锁,在事务结束时会释放

        # 6.6.1. 共享锁 (行锁) Shared Locks

        只能读 不能改

        又称为读锁 (S锁), 多个事务对同一条数据可以共享医保所,都能访问到数据,但是不能修改;

        加锁: Select * from TB_01 where id=1 LOCK IN SHARE MODE

        # 6.6.2. 排他所 (行锁): Exclusive Locks

        又称写锁, (X锁) 只有取得写锁的事务才能对数据进行修改.

        默认超时时间 50秒

        加锁方式:

        • 自动: delete/ update/ insert 默认加上X锁
        • 手动: Select * from TB_01 where id=1 FOR UPDATE

        # 6.6.3. 意向共享锁 (表锁) Intention Shared Locks

        IS锁 , 存储引擎自己维护,无需用户关心

        **帮助理解:**就是一个标识,为了提升加锁的效率, 否则如何判断一张表那条数据被加锁了呢? 全表扫描?

        前提: 一个事务能够成功给一张表加上表锁的前提是, 没有其他事务给这张表的任意一行数据加锁 **如果检查没有锁: ** 任何一个事物加行锁的时候,需要先给这张表加个意向锁, 来标示这张表有锁.

        # 6.6.4. 意向排它锁 (表锁) Intention Exclusive Locks

        IE 锁, 存储引擎自己维护,无需用户关心

        **帮助理解:**就是一个标识,为了提升加锁的效率, 否则如何判断一张表那条数据被加锁了呢? 全表扫描?

        前提: 一个事务能够成功给一张表加上表锁的前提是, 没有其他事务给这张表的任意一行数据加锁 **如果检查没有锁: ** 任何一个事物加行锁的时候,需要先给这张表加个意向锁, 来标示这张表有锁.

        # 6.6.5. 临键锁 (Next-key Lock)

        临键锁包含间隙锁

        解决幻读问题, 对不存在的数据区间进行加锁

        # 6.6.6. 间隙锁 Gap locks

        只有(RR可重复读)才有间隙锁,, Innodb的默认隔离级别

        间隙锁跟间隙锁不冲突, 目的是用于阻塞插入的;

        在插入的时候,会锁定一个区间... 比如当前数据6,7,10的数据,

        • 此时来插入8的数据会锁住 7-10之间
        • 此时来插入11, 会锁住10-无穷大区间

        # 6.7. 锁的算法

        为什么要设置一个锁机制呢?

        答: 为了解决资源竞争问题, 解决事务并发;

        InnoDB的行锁到底锁住了什么?

        通过索引(不分主键索引/普通索引)进行的加锁... 在查询的时候需要指定索引=XX, 不然就会**全表扫描(表锁) **

        • 没索引. Select * from TB_01 where id=1 排它锁(X行锁) , 需要全表扫描找到id=1的数据锁表
        • 有索引. Select * from TB_01 where id=1 排它锁(X行锁) 只会锁住当前行
        • 总结 RR锁表,RC锁行

        # 6.8. 死锁的发送条件

        • 互斥
        • 不可剥夺
        • 形成等待环路 A->B->C->A

        # 6.9. 开启标准监控和锁监控

        • set GLOBAL innodb_status_output=NO
        • set GLOBAL innodb_status_output_locks=NO

        # 6.10. 锁的避免

        • 顺序访问(先操作A再操作B)
        • 数据排序(临键锁,间隙锁)
        • 申请足够级别的锁
        • 避免where条件未命中情况
        • 大事务分解成小事务
        • 使用等值查询而不是范围查询
