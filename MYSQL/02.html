    # 1. 相关名词

    索引、索引覆盖、回表、最左匹配、聚簇索引 执行计划 锁、事务 、MVCC 事务ACID、幻读 读写分离、分库分表、主从复制、MTS(组提交)

    # 2. 索引

    索引的创建跟存储引擎是挂钩的

    1. 索引数据、行数据 存储在磁盘
    2. 将进行查询的时候需要 以页4K的整数倍（磁盘预读）读取到内存
    3. 索引提升效率 减少IO的次数 减少IO的量

    # 2.1. 索引是什么

    是帮助mysql搞笑获取数据的 索引是排好序的数据库结构

    索引是排好序的,但是数据存储在磁盘上是无序的,因为数据是存在磁道上的,过段时间这个数据删了就会替换为别的数据

    • 二叉树
    • 红黑树
    • AVL树
    • Hash表
    • B-Tree (念 B杠树)
    • B+Tree 🔥

    # 2.2. 树结构

    缺点: 随着数据量越大,树越深, 那么IO次数就上来了; 影响数据读取的效率

    为什么影响读取效率? 因为 每一个节点中有且只有两个分支, 把原来二叉树变成多叉树就解决了这个问题,这就是 B-Tree

    # 2.2.1. 二叉树 BST (Binary Seach Tree)

    二叉树本身是无序的, 但是为了提高对于数据查询的效率

    插入数据的时候必须有序, 左子树必须小于根节点, 右子树必须大于根节点

    使用二分查找提高效率

    缺点: 会出现线性问题

    # 2.2.2. AVL树 二叉树平衡树

    经过左旋或者右旋让树平衡起来

    插入效率低, 查询效率高 适用写少读多的情况

    缺点: 为了保证平衡在插入的时候必须要旋转, 用插入时的损失来提高查询性能的提升

    # 2.2.3. 红黑树

    解决了AVL在读少写多情况的不足

    查询询性能和插入性能近似,取得一个平衡

    # 2.2.4. B-Tree

    树结构每一个节点中有且只有两个分支 把原来二叉树变成多叉树就解决了这个问题,这就是 B-Tree

    对应mysql这种场景, B-Tree有自己的局限性;

    B-tree的非叶子节点也要存储数据, 每个页16KB能存储的数据有限 而B+tree的非叶子节点不存储数据

    # 2.3. B+Tree

    只在非叶子节点存储数据, 每个叶子节点都是前后关联的(最底层叶子组成一个双向链表)

    Mysql的B+TREE 一般情况下 3-4层足以,如果超过这个数就要考虑分库分表了

    如果我想要我的b+tree只停留在三层,我应该怎么做?

    • 答案: 选择合适的索引字段类型, 选择占用字节少的字段类型

    # 2.4. Hash 表

    • 优点: 如果是等值查询,那么速度非常快O(1)
    • 缺点: 存在碰撞问题会产生线性问题,不支持范围查询, 对内存的要求比较高

    Memory 的存储引擎就是hash表

    Innodb支持自适应hash, 存储引擎决定到底是用树来存还是hash来存, 人为无法干预

    # 2.5. OLAP, OLTP

    • OLAP: 对应数据仓库(Hive)
      • 联机分析处理, 对海量历史数据进行分析
    • OLTP: 对应数据库(Mysql)
      • 联机事务处理, 要求在很短的时间内返回响应结果

    # 2.6. 磁盘是用多少读多少吗?

    磁盘预读,局部性原理

    # 2.6.1. 局部性原理 (操作系统原理)

    • 时间局部性

      • 在某一段时间内,数据产生聚集倾向

    • 空间局部性

      • 数据和程序都有聚集成群的倾向,同时之前被访问次数多的数据很有可能再次被查询

    # 2.6.2. 磁盘预读

    内存在跟磁盘交互的时候, 一般情况下有一个最小逻辑单元. 这个逻辑单元称之为页.. 每次进行交互的时候由页为单位,一般是4k\8k.. innodb的页大小默认是16K, 每次读取16K的数据

    我们在进行数据交互的时候,可以以页整数倍进行读取

    # 2.7. 索引存在哪里?

    存在内存: 如果断电就丢失

    存在磁盘, 查询数据的时候优先将索引加载到内存中

    # 3. MySQL 的存储引擎

    Innodb , Memory, myisam

    5.1版本之前默认是 myisam

    之后默认是 Innodb

    # 3.1. 什么是存储引擎

    数据在磁盘中的组织形式, 不通的引擎存储方式不一样

    Innodb的存储文件

    • frm是表结构
    • idb是数据+索引 (聚簇索引)

    Myisam

    • frm 表结构
    • MYD 数据
    • MYI 索引

    Innodb 和myisam对比:

    索引不一样

    锁

    • myisam: 支持表锁
    • Innodb: 支持表锁,行锁

    外键: 只有innodb支持 事务: 只有innodb支持

    # 3.2. 聚簇索引 (Innodb)

    聚簇索引就是指表数据和索引是不是存储在一块的

    Innodb只能有一个聚簇索引,但是可以有很多非聚簇索引

    在B+Tree中的存储结构 K: 主键ID V: (行数据)

    Innodb至少有一个聚簇索引, innodb在插入数据的时候必须要包含一个索引的key值, 索引可以是主键,也可以是唯一键....如果没有建立索引会生成一个六字节的ROWID java的int是4字节可以存21E

    # 3.2.1. 回表

    假设有张表有 id, name 两个索引列 SELECT * FROM TB WHERE name='xiaoli';

    那么首先会根据name这个索引查询出对应的id,, 再使用ID索引去查询行的数据 以上操作走了两个B+TREE, 这个现象就叫回表

    # 3.2.1.1. MRR mult_range read

    在进行范围查找的时候, 通过索引字段查询出数据,然后在内存中对主键进行排序 然后通过回表查询数据返回

    # 3.3. 索引覆盖

    假设有张表有 id, name 两个索引列 SELECT id,name FROM TB WHERE name='xiaoli'; 以上情况直接根据name的聚簇索引查询出数据返回即可; 以上情况就叫索引覆盖

    # 3.4. 最左匹配 (适用于联合主键)

    假设有张表有 id, name, age 三个字段 id是主键,(name, age)是组合索引

    • SELECT id,name FROM tb WHERE name='?' and age='?'; ✅
    • SELECT id,name FROM tb WHERE name='?'; ✅
    • SELECT id,name FROM tb WHERE age='?'; ❌
    • SELECT id,name FROM tb WHERE age='?' and name='?'; ✅ (sql优化器)

    # 3.5. 索引下推 (5.7以后特性)

    假设有张表有 id, name, age 三个字段 id是主键,(name, age)是组合索引

    SELECT id,name FROM tb WHERE name='?' and age='?';

    客户端 -> 服务端(server) -> 存储引擎

    没有索引下推: 先在存储引擎中查询 符合name的数据,然后在server端对age进行过滤

    有索引下推: 根据name,age两个条件从存储引擎中获取数据

    # 4. 分布式MYSQL

    # 4.1. bin log (Binary log)

    mysql server的日志, 用于记录DML日志

    主从同步就是靠从数据库读取bin log来实现的

    # 4.2. redo log （中继日志）

    innodb的日志

    # 4.3. undo log

    innodb的日志， 记录事务

    当我操作完某一条DML语句后，记录状态数据。

    # 4.4. 主从复制 （必须开bin log）

    主从复制主要是用于MYSQL的数据同步

    1. 主节点必须开binlog ,在DML语句的时候会写入bin log
    2. 从节点会开两个线程 IO/SQL线程来完成操作
    3. 从节点的IO Thread来读取binlog, 读取后写入redo log
    4. 从节点的SQL Thread来读取 redo log 进行数据重放，写入数据到表中

    # 4.5. 读写分离

    读写分离需要处理 延时问题

    # 4.5.1. 延迟问题

    主节点N个线程在写，从节点就一个线程在写。这个时候会遇到延迟问题

    # 4.5.2. 顺序/随机读取

    主节点DML写binlog的时候是（顺序）的，从库读取到后从redo log中重放数据时写入数据需要对原有数据修改（随机）；以上操作会造成延迟问题

    5.7以后有个MTS机制（组提交），从根本解决延迟问题

    # 4.6. 分库分表

    分片键 雪花ID

    # 5. 事务 ACID

    • 原子性

      • Undo log

    • 一致性

    • 隔离性

      • MVCC、锁

    • 持久性

      • Redo log

        • WAL （write ahead log 预写日志） 数据没有写成功，但是日志存在那么就可以进行重放修复

      • 二阶段提交

    # 5.1. 二阶段提交 - innodb更新数据流程

    ♦️是执行器执行 ♠️是存储引擎执行

    1. ♦️获取数据
    2. ♠️ 数据页是否存在内存中（不存在则加载）
    3. ♠️返回数据
    4. ♦️更改数据
    5. ♦️写入新的数据
    6. ♠️数据更新到内存
    7. ♠️写入 redo log，数据处于perpaer阶段
    8. ♦️写bin log
    9. ♠️提交事务，数据处于commit状态

    # 6. 杂项

    # 6.1. Innodb 表中有几个索引?

    至少一个

    # 6.2. Innodb 如果数据插入慢可能是什么原因?

    1. 有可能是在数据插入的是产生了B+Tree的上移,存储页的分裂机制

    # 6.3. Innodb 主键要不要自增

    在满足业务需求的情况下,最好是自增. 自增的情况下所有的数据都是往后面做累加的,在B+TREE存储的时候不需要对前面的数据进行操作 ..就不会有存储页的分裂过程

    缺点:

    • 数据容易被爬虫爬走
    • 分库分表时不能使用
