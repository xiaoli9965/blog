                      # 1. 为什么要用 MQ

                      • 异步通信 - 提升系统的相应速度
                      • 系统解耦
                      • 流量削峰 - 利用先进先出特性
                      • 分布式事务解决方案 - 可靠消息投递

                      # 2. 使用 MQ 会带来的问题

                      • 提供了系统的复杂性(多加了一个 MQ 服务器)
                      • 数据一致性问题

                      # 3. 各大厂商的 MQ 对比

                      Active MQ RabbitMQ Rocket MQ kafka
                      开发语言 java erlang java scala
                      单据吞吐量 万级 万级 10w 级 10W 级
                      时效性 ms 级 us 级 ms 级 ms 级以内
                      高可用 高(主从架构) 高(主从架构) 非常高(分布式架构) 非常高(分布式架构)
                      功能特性 文档好,各种协议支持较好 并发能力很强,延时低,有管理界面 MQ 功能完备,性能极好,延时低,有管理界面 为大数据准备的,在大数据领域应用广

                      # 4. RocketMQ

                      阿里巴巴开源的产品,2017 年移交给 Apache 维护 经历过双十一的考验,性能方面应该是没啥问题 支持集群,持久化,事务消息 RocketMQ 默认采用长轮询的拉模式， 单机支持千万级别的消息堆积，可以非常好的应用在海量消息系统中 特性:

                      • java 编写

                      • 支持集群

                      • 持久化 (零拷贝/磁盘顺序存储.随机读/页缓存)

                      • 事务消息

                      • NameServer (相当于注册中心)

                      • Netty nio 通信

                      • 启动端口 9876

                      • 默认启动 borker 占用 8G (需修改启动脚本)

                      • 默认启动 NameServer 占用 4G (需修改启动脚本)

                      # 5. 集群架构

                      • NameServer: 起到控制中心,注册中心作用
                      • Broker: 主要做消息处理(消息存储/事务消息回传)
                      • Consumer: 消费者从 NameServer 中获取 Broker 信息建立连接
                      • Producer: 生产者

                      # 5.1. NameServer💥

                      NameServer 是无状态的,NameServer 之间不会进行数据同步
                      在 Broker 启动的时候会将信息上报到所有的 NameServer 中
                      如果新加入了一个 NameServer Broker 则会吧信息再上报一遍给这个新加入的

                      NameServer 并没有提供类似 Zookeeper 的 watcher 机制， 而是采用了每 30s 心跳机制。

                      # 5.1.1 NameServer 心跳机制

                      • 单个broker和所有NameServer保持心跳请求,间隔30s(心跳信息包含当前 Borker 的所有 TOPIC 信息)
                      • nameServer会反查Borker的心跳信息,如果某个 Borker 在两分钟内没有发送心跳,则该 broker 下线,调整Topic和broker的关系
                      • NameServer 压力不会太大，平时主要开销是在维持心跳和提供 Topic-Broker 的关系数据,但有一点需要注意，Broker 向 Namesrv 发心跳时， 会带上当前自己所负责的所有 Topic 信息，如果Topic个数太多（万级别），会导致一次心跳中，就Topic的数据就几十M，网络情况差的话， 网络传输失败，心跳失败，导致 Namesrv 误认为 Broker 心跳失败。

                      # 5.2. Broker

                      Borker 是分主(Master)和从(Slave)的
                      主节点: 主要处理写操作(主要处理 producer 的信息) 从节点: 主要处理读操作(主要处理 Consumer 的信息)

                      • 通过BrokerName去区分同一组,组内通过 BrokerId 区分主从关系(brokerId为0代表主节点,非0是从)
                      • 一个主可以包含有多个从节点,一个从只能属于一个主
                      • 主从之间是需要进行数据同步(异步/同步)

                      # 5.2.1 TOPIC

                      Broker 上存 Topic 信息，Topic 由多个队列组成，队列会平均分散在多个 Broker 上。Producer 的发送机制保证消息尽量平均分布到 所有队列中，最终效果就是所有消息都平均落在每个 Broker 上。

                      • 每个主题可设置队列个数，自动创建主题时默认4个

                      # 5.3. Producer / Consumer

                      • Producer/Consumer 随机和NameServer中的节点建立一个长连接,定期(30s)从 NameServer 中获取 TOPIC 路由信息
                      • Producer/Consumer会定期(30s)向Broker中的Master节点发送心跳(为啥要发心跳? 因为消息有两种模式 pull/push)
                      • 在发送消息时通过 TOPIC 判断投递到那个 Broker 中,然后直接投递
                      • Consumer 既可以从 Master 订阅消息,也可以从 slave 订阅消息
                      • Producer / Consumer完全无状态,可集群部署
                      • Broker会每10s扫描一次 Producer / Consumer,如果发现某个Producer / Consumer2 分钟没有发送心跳就断开

                      Producer:

                      • 失败默认重试 2 次
                      • 在事务消息机制中，如果发送消息的 producer 在还未 commit/rollback 前挂掉了，broker 会在一段时间后回查ProducerGroup 里的其他实例，确认消息应该commit/rollback

                      # 5.4. 集群模式 1 - 单 Master(单机)

                      只启动一个 Mater 节点,严格的来说算是单机模式

                      # 5.5. 集群模式 2 - 多 Master

                      集群中没有 Slave 节点,都是 Master 节点

                      • 这种方式配置简单,单个 Master 节点宕机对集群无影响
                      • 单节点宕机期间,这台机器未被消费的消息在恢复之前不可订阅,会有窗口期

                      # 5.6. 集群模式 3 - 多 Master 多 Slave(异步)💥

                      HA 采用异步的方式进行同,会有消息延时(毫秒级) 当主收到消息后直接返回消息给生产者,然后进行落库从同步

                      • 优点:可用性高
                      • 缺点:主宕机的话 会有毫秒级的消息丢失,主宕机消费者依旧可以从从节点获取消息

                      # 5.7. 集群模式 3 - 多 Master 多 Slave(同步)💥

                      当主收到消息后进行落库,从同步 ,然后返回消息给生产者

                      • 优点:可用性高
                      • 缺点:
                        • 性能比异步低 10%左右
                        • 主宕机后从无法自动升级为主(目前版本不支持)
                        • 同步复制会增大系统延时降低系统吞吐量

                      # 6. RocketMQ Java 客户端

                      • 客户端发送消息有负载均衡
                      • 客户端内存中保存着当前所有的服务器列表，每次发送都切换一台服务器发送消息，使得每台服务器接收的消息量尽量均衡，避免热点问题。
                      • 消费者端负载均衡集群消费模式下，平均消费该 Topic 的所有队列。可以通过参数调整消费模式

                      # 6.8. 两种消费模式

                      通过 comsumer.setMessageModel(MessageModel.CLUSTERING);设置
                      MessageModel.BROADCASTING 广播 一条消息会被 ConsumerGroup 里的所有实例消费。
                      MessageModel.CLUSTERING 均衡(默认模式) ,一条消息只会被 ConsumerGroup 里的一个实例消费,但可以被多个不同的 ConsumerGroup 消费

                      # 6.9. 消费者拉取的方式 (PULL/PUSH)

                      • PULL 拉取: 定时向 Broker 发发送请求获取数据 DefaultMQPullConsumer
                        • 可以有效的避免给Broker造成巨大的压力
                        • 需要用户自己处理遍历 MessageQueue、保存 Offset,所以 PullConsumer 有更多的自主性和灵活性。
                      • PUSH 推送: 消费者和 Broker 建立一个长连接状态 DefaultMQPushConsumer
                        • push也是用pull实现的,采用的是长轮询方式 broker 每 5 秒钟检测一次是否有消息,有消息就传给你
                        • push 模式在收到消息请求后,如果队列里没有消息并不急于返回,在经过三轮的检查后才会进行返回(每轮5s)
                        • 长轮询的主动权还是掌握在 Consumer 手中,Broker 即使有大量消息积压，也不会主动推送给 Consumer

                      # 6.14. RocketMQ 顺序消息 🔥

                      • 全局消息顺序: (没有必要)broker 中所有队列要求按顺序消费
                      • 局部消息顺序: 😈 只要保证某一个队列按顺序消费

                      如何保证局部消息顺序呢?

                      简单的理解就是同一个业务的消息发到同一个队列就行

                      顺序消息发送:

                      Rocket 默认是随机发送消息到队列中

                      如果要实现顺序消息那么只能指定发送到指定的一个队列中,在 send 方法中第二个参数传入 MessageQueueSelector 中指定队列下标

                      顺序消息接收:

                      客户端会根据每一个队列开启一个线程进行消费,当消息都在一个队列的时候就会在线程中顺序消费

                      # 6.15. RocketMQ 延时消息 🔥

                      在 RocketMQ 中延时的时间已经规定好了一些枚举,不能自定义时间 messageDelayLevel = "1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h" 延时消息会直接发送到队列中,过一段时间后会被消费

                      # 6.16. RocketMQ 失败重试 处理策略

                      消息重试会造成消息的堆积

                      默认情况下衰减重试是 16 次(和延时队列的时间一个时间 messageDelayLevel = "1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h),延迟等级为 3~18。 十六次之后还是失败就会将消息丢该死信队列(DLQ)

                      处理失败消息可以考虑:

                      可以判断其衰减次数然后实例化到表中,进行相应后续逻辑

                      # 7. RocketMQ 死信队列(DLQ)

                      Dead-Letter-Message 死信队列是 MQ 中的一个通用设计

                      当消息消费失败(重试到达最大次数),就会吧消息丢到死信队列中

                      死信队列的保存期限是三天,三天后自动删除

                      # 8. RocketMQ 批量消息

                      批量消息中,一次发送建议不要超过 4M

                      # 9. RocketMQ 消息过滤

                      • 根据 TAG 进行过滤,在消息发送的时候指定消息 TAG,消费的时候指定消费的 TAG
                      • 根据简单 SQL 语法进行消息过滤 比如 A > 5 ADN B = 'abc'

                      # 10. RocketMQ 事务消息(4.3+)🔥🔥🔥

                      这个机制可以用于分布式事务 ,TransactionMQProducer

                      • 生产者发送事务消息发送的时候对消费者是不可见的(半消息)
                      • 当生产者收到 broker 的成功响应之后执行本地的事务
                      • 本地事务执行完成后再去对半消息进行Rollback或commit操作
                      • 当消息 commit 之后消息就可以被消费者消费了
                      • 当消息 rollback 之后半消息会被删除
                      • 如果超时没有对半消息进行处理,MQ 服务器会对消息进行回查,然后触发 rollback 或 commit

                      事务消息的三个状态:

                      • 提交(commit) :允许消费者消费
                      • 回滚(rollback) :消息将被删除,不允许消费
                      • 中间(unkown) :MQ 服务器会进行会查消息在决定 rollback 或 commit

                      # 11. RocketMQ 消息的幂等性 🔥🔥

                      出现重复消息的情况:

                      • 生产者导致, 可能由于网络原因导致生产者没有收到服务端回复导致消息重复发送了
                      • 消费方导致, 由于网络原因导致它没有给 MQ 服务器做一个正确的回应,这个时候可以回产生消息的重复投递
                      • 负载均衡导致, 如果消息的消费方,消费策略是负载均衡.如果当消费者接收到了消息后由于宕机导致服务器 rebalance,此时可能会导致消息重复

                      如何保证幂等性:

                      不能用消息ID(rocketMq 不保证消息ID唯一性),最好的办法是使用户业务 ID 作为幂等判断

                      # 13. RocketMQ 消息存储(刷盘)

                      # 13.1. 消息存储 - 顺序读写

                      **顺序读写:**目前高性能的磁盘(SSD) 顺序读写的速度达到了600M/s,随机读写仅 100kb/s

                      # 13.2. 消息发送 - 零拷贝技术

                      零拷贝技术对文件大小有要求,要求文件大小 1.5G-2G , 正式因为这个条件 Rocket 将 commitLog 大小设置为 1G 的原因

                      # 13.3. 刷盘机制 (同步/异步)

                      • 同步: (更安全,相比速度慢10%)当收到消息后,先写入磁盘,后返回确认消息
                      • 异步: (效率更高)当收到消息后,直接返回消息确认,同时进行刷盘

                      # 14. 存储结构

                      • CommitLog: Rocket 所有的 topic/QueueId/message 都会被存储到 commitLog 文件中 commitLog 文件是系统自动生成的(大小1G)
                      • ConsumerQueue: 存储消息在 CommitLog 的索引,用于加速 CommitLog 消息的查找
                        • 每一个消息队列都会生成一个对应的ComsumerQueue
                        • 主要是记录消息的偏移量
                        • 这个 ConsumerQueue 如果丢了可以通过 CommitLog 问件反向生成
                      • IndexFile: 为了提供一些范围查询,通过 indexFile 来查找消息不影响发送和消费消息的主流程
