                                                                            # Redis 面试题

                                                                            # 1. 🤣Memcached 和 Redis 的区别

                                                                            • memcached 一般不持久化(可以持久化) ,Redis 可以持久化
                                                                            • memxached 只能存储简单的字符串, Redis 支持多种数据类型
                                                                            • memcached 实现:多线程+锁 , redis 实现:单线程+IO 多路复用

                                                                            # 2. Redis 单线程吗

                                                                            redis 在处理数据的时候用的单线程,实际上整个服务是多线程的

                                                                            用到多线程的地方:

                                                                            • RDB 时 FORK子线程
                                                                            • 主从同步时的异步同步 PSYNC
                                                                            • 关闭服务的时将内存刷新到磁盘的操作
                                                                            • LazyFree机制(4.0 特性)
                                                                              • UNLINK 异步删除 Key
                                                                              • FLUSHDB ASYNC-清空本库的数据
                                                                              • FLUSHALL ASYNC-清空所有数据库

                                                                            # 3. Redis 单线程为什么性能这么高

                                                                            • 协议很简单 就是 cmd key Val 不像 HTTP 那样又是请求体,请求头的
                                                                            • 基于内存操作
                                                                            • pipline 机制,将多个指令一起发送到服务器执行,减少了网络往返的时间
                                                                            • NIO 多路复用

                                                                            # 4. 为什么 Redis 的操作是原子性的，怎么保证原子性的? 多个命令在并发中也是原子性的吗

                                                                            1.Redis 的操作之所以是原子性的，是因为 Redis 是单线程的。
                                                                            2.多个命令在并发中并不保证原子性,可以使用事务方式解决;

                                                                            # 5. 单 redis 实例支持并发读写吗？例如客户端 A 写入一个很大的数据， 需要耗时几秒， 期间客户端 B 能否读数据？还是需要等待客户端 A 的操作完成才能读到数据

                                                                            redis 有多路复用机制,支持多个请求并发请求..但是在写数据时需要等 A 写完才能轮到 B 写

                                                                            # 6. Redis 集群大小,可以支持多少个分片

                                                                            redis 有 16834 个槽,理论上支持 16834 个分片,但是官方建议最大 1000 个实例, 因为 redis 集群有个总线..如果真的有 1W6 千个节点,那么每个节点总线的数量就是(1W6-1)\*2个,这样的话每个节点的消息量就会非常大

                                                                            # 7. 集群节点间怎么通信的

                                                                            每个节点都有一个 TCP 端口,每个节点使用 TCP 连接其他节点..故障转移的方式和哨兵模式类似.

                                                                            # 8. ask 和 moved 的区别

                                                                            重定向包括两种情况

                                                                            • 若 slot 不属于当前节点,redis 会返回 moved
                                                                            • 若当钱节点正在处理 slot 迁移,如果当前请求对应的 key 已经被迁移至新节点,就会返回 ask 告诉客户端本次请求重定向

                                                                            # 9. 集群中数据倾斜,访问倾斜问题怎么处理

                                                                            倾斜会导致及群众某些节点访问量增大,压力大. 解决方案:

                                                                            • 提前预测,那些 key 会是热点,在设计过程中规避
                                                                            • 后期 slot 迁移,尽量将压力分摊(slot 调整有两种方式 自动:rebalance和手动reshard)

                                                                            # 10. 集群中的读写分离怎么操作 🙄

                                                                            • 默认情况下,redis 集群中读和写都会重定向到 key 槽所在的节点上
                                                                            • 可通过 readonly 设置当前连接可读,通过 readwrite 可以配置从节点可读.(会导致数据不一致)

                                                                            # 11. 💥 租房平台面试题:去重方案

                                                                            有一张访问信息表,有 地址,房间号等其他字段... 在添加的时候是如何判断重复的? (表数据 100w)

                                                                            解决方案:

                                                                            布隆过滤器(省内存.会有误差) Redis 有个 bloom 过滤的插件,还有一个专用的 JReBloom 包,可以实现布隆过滤器 通过 bf.add命令添加 添加的时候可以指定误判率 https://github.com/RedisBloom/RedisBloom

                                                                            布隆过滤器只能添加元素 不能删除元素

                                                                            # 12. 💥 限流 (令牌桶/漏斗桶)

                                                                            • 令牌桶: 资源有限,先到先得
                                                                              • 基于 List 的数据结构 创建 N 个数据量的令牌.取的时候通过 leftPop 获取令牌
                                                                            • 漏斗桶: 资源无限,限时限量发放 业务场景: 限制同一个用户,同一个功能, 10 分钟内只能访问 60 次 方案 1: 设置一个 10 分钟过期的 Key 方案 2: zset, 每次操作插入一条数据(十分钟过期),吧当前时间作为 score

                                                                            # 13. 分布式锁 (最优方式使用 ZK)

                                                                            背景:多个服务同时操作同一个资源,协调资源操作

                                                                            • 2.8 以前通过 set + expire两个命令 实现带过期的锁
                                                                            • 2.8 以后 通过 setnx + expire 或 setex +expire 来实现带超时的锁. 操作执行完毕后通过 set 的 value 值判断是否是通一把锁来删除锁

                                                                            为什么说他不是最优方案:

                                                                            • 主从环境下:,set 锁到主的时候成功了,客户端 1 获取到了锁 ,然后主挂了故障转移从当上了主(数据还未完成同步主就挂了 锁丢失). 这个时候客户端 2 来获取锁也能成功获取到锁
                                                                            • 锁删除问题: 如果设置了锁过期,客户端 1 获取了锁,当锁时间到了,客户端 1 还没有执行完(执行时间较长),然后客户端 2 获取到了新的锁,这个时候客户端 1 执行完了将锁删除 ❌...这种时候需要编写逻辑进行处理(麻烦)

                                                                            # 14. 什么是缓存击穿,穿透,雪崩? 如何预防

                                                                            • 缓存击穿: 缓存中有当前 KEY 值,当客户端访问无法获取到这个缓存(比如 KEY 正好过期)
                                                                            • 缓存穿透: 缓存不缓存 null 值,客户端一直请求一个数据库不存在的值,此时请求不会走缓存直接进入数据库
                                                                            • 缓存雪崩: 缓存雪崩就是击穿或穿透的连锁效应, 高并发场景下,因为一个 KEY 没有缓存导致请求直接到达数据库,导致数据库压力大宕机的情况;

                                                                            预防方式:

                                                                            • KEY 的过期是时间设置不一样,最好弄一个随机值
                                                                            • 对于单个热点 KEY,当其失效刷新时加上互斥锁,使只有一个请求去刷新缓存. 避免大量请求直接请求到数据库
                                                                            • 对于服务不可用/恶意请求/网络不可用时
                                                                              • 把查到数据库为 null 的值也缓存起来(加上过期时间),使恶意攻击时也会走缓存
                                                                              • 使用布隆过滤器排除一些不存在的 KEY,直接进行拦截 (布隆过滤器有一个确定,添加了不能删除)

                                                                            # 15. 项目中缓存是如何使用的？为什么要用缓存？缓存使用不当会造成什么后果

                                                                            缓存如何使用: 存储一些热点数据, 读多邪少的数据 为啥要使用缓存: 提升服务响应速度,分担数据库压力 使用不当会造成什么后果: 缓存雪崩效应

                                                                            # 16. 为什么 redis 单线程却能支撑高并发

                                                                            • 协议很简单 就是 cmd key Val 不像 HTTP 那样又是请求体,请求头的
                                                                            • 基于内存操作
                                                                            • pipline 机制,将多个指令一起发送到服务器执行,减少了网络往返的时间
                                                                            • NIO 多路复用

                                                                            官方的测试数据通过 pipline 写 string 请求支持没秒 50W 数据读写

                                                                            # 17. redis 都有哪些数据类型？分别在哪些场景下使用比较合适

                                                                            • String 场景:
                                                                              • 缓存对象通过(JSON 序列化) set key value
                                                                              • 单个值缓:如分布式锁 setNx setEx ,分布式 sessionc 存储 TOKEN
                                                                              • 多个值缓存: MSET,一次设置多个 key value
                                                                              • 计数器: INCR
                                                                            • Hash 场景:
                                                                              • 存储对象 ,不用 JOSN 序列化缓存,也阔以直接对 KEY 中某个字段直接进行操作
                                                                            • List 场景:
                                                                              • 实现队列功能 LPUSH,RPUSH,LPOP,RPOP
                                                                              • 阻塞队列实现:BLPOP,BRPOP
                                                                              • 实现栈的功能: 后进先出
                                                                              • 返回区间值 LRANGE 1 10
                                                                            • SET/ZSET 场景:
                                                                              • 阔以实现去重存储
                                                                              • 还阔以实现取交集/并集/差集的功能

                                                                            # 18. 在集群模式下，redis 的 key 是如何寻址的？分布式寻址都有哪些算法？了解一致性 hash 算法吗

                                                                            redis 中的一致性 hash 算法就是 对 key 进行 hash 取模得出槽 slot 的位置,然后到槽所在的机器上进行存储

                                                                            在 JAVA 客户端集群连接下,,客户端会获取到集群环境的 slot 分布情况,在客户就先将 key 进行寻址 slot,然后直接到相应集群进行存储

                                                                            # 19. 如何保证缓存与数据库的双写一致性 又名:(redis 的并发竞争问题是什么？如何解决这个问题)

                                                                            什么是双写: （1）线程 A 更新了数据库 （2）线程 B 更新了数据库 （3）线程 B 更新了缓存 （4）线程 A 更新了缓存

                                                                            （1）请求 A 进行写操作，删除缓存 （2）请求 B 查询发现缓存不存在 （3）请求 B 去数据库查询得到旧值 （4）请求 B 将旧值写入缓存 （5）请求 A 将新值写入数据库

                                                                            延时双删策略: （1）先淘汰缓存 （2）再写数据库（这两步和原来一样） （3）休眠 1 秒，再次淘汰缓存 解决方案：采用双删除策略。写请求先删除缓存，再去更新数据库，等待一段时间后异步删除缓存。这样可以保证在读取错误数据时能及时被修正过来。

                                                                            两次更新策略: （1）更新缓存 （2）再写数据库 （3）再新缓存 这种方案保证了读写的一致性，但由于读请求等待写请求的完成，会降低系统的吞吐量。

                                                                            # 20. 了解 redis 事务的 CAS 方案吗

                                                                            就是更新缓存的时候进行锁住,同时只允许一个请求进行更新操作,更新完后释放锁 (解决缓存雪崩问题)

                                                                            # 22. 介绍一下 HyperLogLog

                                                                            HyperLogLog 是在 2.8 版本加入的一个数据类型,主要用来做基数统计算法的

                                                                            优点: 在数据体量非常大时,计算基数所需的空间是固定的,并且很小只需要 12KB 只要花费 12KB 内存就可以计算 12^64-1 的数据量

                                                                            什么是计算基数: 就是计算不重复的元素 {1,2,3,3,4,5,5,8} -> {1,2,3,4,5,8}

                                                                            为什么用它: 如果计算 1 亿个基数,大约需要 1E/8/1024/1024 =12M 内存,如果不用 hyperLogLog 统计 10000 个对象就差不多要 120G

                                                                            # 23. Redis 分区有什么缺点

                                                                            优点 性能高 缺点 集群分片很难玩(请求倾斜问题/KEY 落在不同分片问题/动态扩容/容缩非常复杂)

                                                                            # 25. 修改配置不重启 Redis 会实时生效吗

                                                                            可通过 CONFIG SET 命令进行修改,无需重启

                                                                            # 26. Redis持久化方式

                                                                            Redis持久化方式

                                                                            • RDB : 快照模式 , 每间隔一段时间fork一个子线程进行数据快照保存到 .RDB的文件中
                                                                            • AOF : 写日志模式 , 默认每间隔一分钟就行一次日志追加

                                                                            两种方式的优缺点:
                                                                            RDB 优点

                                                                            • 可用用来做数据库定时备份
                                                                            • 备份数据占用磁盘空间相对AOF较小
                                                                            • 数据库启动时恢复数据速度比较快

                                                                            RDB缺点

                                                                            • 存在数据丢失问题,因为快照是有一定周期的,在还没有达到下一次快照周期是服务器宕机会导致数据丢失
                                                                            • 系统默认的设置是在规定周期内根据改变数据的数量来进行触发快照的,在数据量较大的情况下这种备份机制会导致服务器压力过大 (数据改变频繁时,每分钟都要进行数据库全量快照然后写入磁盘)
                                                                            • fork子线程的时候回占用一些服务器内存来进行数据快照, 会消耗服务器内存

                                                                            AOF优点:

                                                                            • 数据丢失风险较小, 因为1分钟一次写LOG,即使数据丢失也就丢失一分钟的数据
                                                                            • AOF有一个日志重写方法,可以将部分没必要的日志进行合并 (比如连续的三个set同一个K ,可以直接取最后一个操作)

                                                                            AOF缺点:

                                                                            • 启动时因为要从日志中恢复数据,所以启动速度比较慢
                                                                            • 日志的方式比快照的方式更占用磁盘空间

                                                                            # 26.1. RDB

                                                                            RDB快照模式 - 默认采用子进程方式

                                                                            有两种实现方式 一个阻塞(save命令) 一个异步非阻塞(bgSave命令)
                                                                            • save命令会导致Redis整个操作阻塞
                                                                            • bgsave是直接fork一个子线程进行快照,快照到一个零时文件然后覆盖原有快照文件

                                                                            WARNING

                                                                            系统默认触发bgsave的条件

                                                                            • 15分钟一个数据改变
                                                                            • 5分钟100个数据改变
                                                                            • 1分钟 1W个数据改变

                                                                            # 26.2. AOF

                                                                            • AOF默认方式是 1分钟写一次LOG
                                                                              • 这种模式默认会有一个子线程进行写LOG,,还有一个缓冲区进行记录数据的变化 , 主线程主要负责数据存储,子线程定期读取缓冲区的数据进行刷盘

                                                                            AOF支持的方式

                                                                            • always : 每一次数据改变都会写入日志
                                                                            • everysec : 每一分钟进行一次写入日志 (默认方式)
                                                                            • no : 就是根据系统自动判断写日志方式

                                                                            AOF模式会导致日志文件巨大,如何解决?

                                                                            系统有一个日志重写功能, 就是讲日志部分命令进行合并,减少了日志文件大小,节约了数据库启动时间

                                                                            # 27. Redis主从复制

                                                                            主从环境下从库默认是可以写入数据的,但是不会向上同步,,一般会开启从库只读配置,,
在配置文件中配置   slave-read-only 为yes

                                                                            • 主从复制开启:

                                                                              • 方式1 : 配置文件中配置slaveof (需要重启)
                                                                              • 方式2 : 客户端直接输入命令(不需重启,重启后失效)

                                                                            • 主从复制模式:

                                                                              • 一主多从 : 这种所有的从库都直接备份主库的数据 主压力较大
                                                                              • 主从从 : 从库1备份主库数据, 从库2备份从库1 , 这种方式通过从库来RDB备份是很好的方式

                                                                            • 主从复制的原理:

                                                                              • 建立主从关系之后,从库会想住发起同步命令
                                                                              • 主库接到命令后开始RDB快照(快照期间改变的数据会缓存起来)
                                                                              • 主将快照文件和缓存一并发给从
                                                                              • 从库完成同步
                                                                              • 后续发生的数据改变命令主库会将命令发送给从库,从而保证数据一致性

                                                                            # 31. 遇到过缓存穿透吗? 遇到过缓存击穿吗? 如何避免缓存雪崩

                                                                            原则: 把有效的请求交给数据库,无效的请求过滤掉..即便放大前置环境的复杂度和成本

                                                                            • 穿透: 直接访问到了没有的数据,导致每次都请求数据库..
                                                                              • 对nullkey进行缓存, 高并发情况下需要加锁,先取锁在查询缓存
                                                                              • 使用布隆过滤器
                                                                            • 击穿: 请求了 过期的key活没有被缓存的key,导致请求直接走到数据库
                                                                              • 处理方法:大家抢锁,没抢到的sleep,抢到的去DB,然后更新redis,sleep的返回到第一步;
                                                                                • 线程sleep的时候是bloking的轻量级进程,不参与CPU及内核调度,做好线程池扩缩容就可以了
                                                                              • 锁要用单机的,要有排他性

                                                                            如何避免缓存雪崩:

                                                                            1. 参照击穿处理办法
                                                                            2. 击穿锁可以是分片的,因为每个key互不干扰

                                                                            核心思想,避免DB的无效/重复请求

                                                                            # 39. Redis的用途定义

                                                                            • 如果定义为缓存服务器的话,需要有内存淘汰策略. 那么将不适用于做分布式锁
