                                    # 2. 内存淘汰策略

                                    淘汰策略用于移除在内存中但是又不用的数据(释放内存空间) 为了防止内存溢出

                                    内存淘汰主要分为 5 种类型：

                                    • 不淘汰: 内存溢出就报错 noeviction
                                    • LRU: volatile-lru , allkeys-lru 所有 key 或者快过期的 key 中淘汰最不常用的
                                    • LFU(4.+): volatile-lfu , allkeys-lfu 所有 key 或者快过期的 key 中淘汰最少使用的
                                    • 随机淘汰: volatile-random ,allkeys-random 所有 key 或者快过期的 key 中随机淘汰
                                    • TTL: volatile-ttl 优先淘汰快过期的

                                    策略 解释
                                    🔥volatile-lru 在所有设置了过期时间的 key 中,删除最不常用的数据.
                                    🔥allkeys-lru 查询所有 key 种最不常用的数据进行删除
                                    volatile-lfu (4+版本) 在设置了过期时间的 key 中淘汰快过期最不常用的
                                    allkeys-lfu (4+版本) 从所有键中淘汰最不常用的
                                    volatile-random (不可控)在已设置了超时时间的 key 中进行随机删除
                                    allkeys-random (不可控)在所有 key 中随机删除
                                    volatile-ttl 查询所有设置了超时的 key 中,删除马上要过期的 key
                                    noeviction 不淘汰,如果内存溢出了就报错返回

                                    # 1. LRU (Least Recently Used) 最近使用

                                    中心思想: 最近被使用过,也行未来被使用的记录会更高

                                    底层就是一个固定长度的双向链表，当新值进来的时候就在头部插入，如果超出长度就把尾部剔除

                                    # 2. LFU (Least Frequently Used) 最不常用

                                    中心思想: 如果一个数据在最近一段时间很少被访问到，那么可以认为在将来它被访问的可能性也很小。因此，当空间满时，最小频率访问的数据最先被淘汰

                                    类似于计数器的设计，淘汰计数器最少 key

                                    # 3. 过期 KEY 处理策略

                                    # 3.1. 主动处理

                                    每 100ms 秒扫描一次所有带过期的 KEY ,如果本次淘汰的 KEY 超过过期 KEY 总数的 25%就会再次进行扫描,直到所有 KEY 都过期

                                    # 3.2. 被动处理(惰性删除)

                                    当请求到这个 KEY 发现这个 KEY 已经过期了就会将其删除

                                    # 4. RDB 和 AOF 时对于过期 KEY 的处理

                                    • AOF 在恢复的时候会判断这个 KEY 是否带时间是否已经到期,如果到期就会再后面加上一个 DELETE 命令
                                    • RDB 快照时不会存入已过期的数据,,如果存入的 KEY 过期了就会执行主动处理方式

                                    # 5. 各各数据类型的字段长度限制 💥

                                    • String K 和 V 最大 512M
                                    • List 元素最多 42 亿个(int 最大值 2^32次方-1)
                                    • set/zSet 元素最多 42 亿个(int 最大值 2^32次方-1)
                                    • hash 元素最多42 亿个(int 最大值 2^32次方-1)

                                    # 6. 内存压缩 (内存优化手段)

                                    可以通过配置文件指定各各数据类型的 字段最大长度,元素最大个数 例:hash-max-zipmap-entries 512, list-max-ziplist-entries512 当个数和长度在这个范围内就会进行压缩,如果超过指定范围就不压缩(因为会消耗 CPU)

                                    何时执行快照?

                                    # 8. AOF 写日志

                                    • appendonly yse: 开启 AOF
                                    • appendfsync allays: 每条修改/新增都进行写 APF (很安全,很消耗性能)
                                    • appendfsync everysec: 没秒进行写 AOF (由于有磁盘缓存,并不是真正的没秒写)
                                    • appendfsync no: 不主动写 AOF
                                    优点 缺点
                                    备份粒度更小 安全,容灾 AOF 文件体积较大
                                    易读,可以修改 🙄性能消耗比 RDB 高,数据恢复比 RDB 慢

                                    过程:

                                    • 将当前内存中的数据以一定的格式存储写入到 temp.aof 文件中
                                    • 当写完后通过原子操作 将原有的 AOF 文件改名 ,再将 temp.aof 改为 appendonly.aof
                                    • 后续发生的改变会被追加到 aof 文件中

                                    结论: 不管怎么方式都有丢失数据的可能性,除非用关系数据库进行原子操作

                                    RDB: 快照方式天生就不保证数据的安全,保证效率

                                    主从复制: 异步的存在一定的窗口期, 会受到 网络/服务器性能的影响导致数据丢失

                                    # 12. 前提知识点

                                    • redis 默认使用异步复制. 同步过程是非阻塞的
                                    • 一个 master 可以拥有多个 slave
                                    • slave 也可以有下级 subSlave
                                    • slace 初次同步需要删除旧数据,加载新数据 (这个过程会阻塞新请求)

                                    # 13. 主从同步过程

                                    # 13.1. RDB 主从同步过程

                                    • slave 节点发送 PSYNC 同步请求
                                    • Master 节点收到请求后 fork 一个子线程进行快照传输给从节点

                                    # 13.2. AOF 主从同步过程

                                    1. Master 节点会给每个 Slave 节点分配一个 ReplicationID ,后续同步过程 slave 会带上这个 ID
                                    2. Master 节点会为每个 slvae 节点创建一个 缓冲区, 每次同步 slave 节点会带上 offset 偏移量,然后主节点根据缓冲区进行传输后续增量数据
                                    3. 如果 Slave 节点传的 ReplicationId 在主节点中没有记录(说明是新上从节点),就会进行 RDB 传输给 Slave 然后为这个节点创建一个 缓冲环,后续通过缓冲还同步增量
                                    • slave 节点发送 PSYNC 同步请求 (请求时带上 replicationId和 Offset)
                                    • Master 节点收到请求后判断同步源是不是当前 Master 节点后(根据 replicationId 判断),根据 offset 进行同步
                                    • 同步源如果不是是当前 Master 节点,则进入全量同步;否则异步生成 RDB 传到 Slave 节点加载到 Slave 节点内存
                                    • 同步源如果是当前 Master 节点,则根据 Offset 偏移量进行异步传输

                                    # 14. redis 有盘同步 / 无盘同步

                                    • 硬盘复制: 在同步的时候会创建一个线程进将数据写到磁盘(RDB/AOF) ,然后通过网络将数据传送给从及节点
                                    • 无盘复制: 主节点创建一个新进程,子进程之间吧 RDB 文件写到从站的套接字中(不需要硬盘) 在硬盘速度低于网络带宽的时候很适用

                                    # 17. 什么样的数据适合缓存

                                    • 访问评率高
                                    • 读多写少
                                    • 一致性要求低

                                    例如: 用户信息,在异构数据同步时经常一个字段只记录用户的 ID,但是同步时需要一些用户的其他数据(可能需要查表三四次),这个时候就用缓存可以极大提升效率

                                    # 18. Zset 的跳表（Skip list)

                                    跳表可以理解成二分查找法的链表(空间换时间)(因为二分查找法的底层依赖数组随机查找的特性,所有链表无法实现二分查找)

                                    为链表建立索引

                                    每两个结点提取一个结点到上一级，我们把抽出来的那一级叫作索引或索引层

                                    查找过程 ,通过最顶级的索引一级一级向下查找,可以大大缩短查找时间.

                                    1.从图中我们可以看出，原来没有索引的时候，查找 62 需要遍历 62 个结点，现在只需要遍历 11 个结点，速度是不是提高了很多？

                                    2.所以，当链表的长度 n 比较大时，比如 1000、10000 的时候，在构建索引之后，查找效率的提升就会非常明显。

                                    redis zset 的底层使用有序集合实现的跳表

                                    • 查找/修改: 通过最上级的索引一级一级的查找,找到后进行操作 比原有链表节省很多时间
                                    • 删除: 查找到后直接删除
                                    • 索引更新: 通过随机函数

                                    # 18.1. 高效的动态插入和删除

                                    我们知道，在单链表中，一旦定位好要插入的位置，插入结点的时间复杂度是很低的，就是 O(1)。

                                    这里为了保证原始链表中数据的有序性，我们需要先找到要插入的位置，这个查找操作就会比较耗时。

                                    # 18.2. 跳表索引动态更新

                                    当我们不停地往跳表中插入数据时，如果我们不更新索引，就有可能出现某 2 个索引结点之间数据非常多的情况。极端情况下，跳表还会退化成单链表

                                    在插入数据的时候 通过一个随机函数，来决定将这个结点插入到哪几级索引中，比如随机函数生成了值 K，那我们就将这个结点添加到第一级到第 K 级这 K 级索引中。

                                    # 18.3. 跳表是不是很浪费内存

                                    比起单纯的单链表，跳表需要存储多级索引，肯定要消耗更多的存储空间。所以我们需要额外再用接近 n 个结点的存储空间。

                                    如果非要节省空间的话可以将原有的每两个节点抽取一个作为索引的策略调整为每 3 个/5 个/10 个

                                    # 18.4. redis zset 底层为什么不适用红黑树实现呢

                                    跳表 红黑树
                                    插入 快 O(logN) 快O(logN)
                                    删除 快 O(logN) 快 O(logN)
                                    查找 快 O(logN) 快 O(logN)
                                    区间查找 支持 不支持

                                    可以看出跳表的特性和红黑树一样,他俩完全可以相互替代

                                    因为红黑树无法实现区间查找(如查找下标[100-500]的值)

                                    # 19. bitmap 位图

                                    可以用来实现 微信点赞功能

                                    BitMap 本质是还是一个 String,它将值转换成一连串的 2 进制数字进行存储,可以实现二进制码级别的操作 位图的最大存储长度是 42E(int 最大值)=512M
                                    例如:set xiaoli a a=1100001 可以直接通过 setbit xiaoli 7 0 将其二进制值进行改变为1100000

                                    实现点赞思路:

                                    好处省内存,只需要以一个 bit 位的容量进行记录

                                    • 以朋友圈 ID 作为 KEY (值的长度会自动扩容,扩容的填充是0)
                                    • 如果用户 ID 为 100 的点赞了朋友圈就在第 100 位 set 为 1
                                    • 取消点赞就相应 ID 位置赋值为 0
                                    • 统计点赞就通过 bitcount 统计

                                    # 21. HyperLogLog

                                    转门用来做统计的,可以用于计算访问量

                                    HyperLogLog 是在 2.8.9 版本加入的一个数据类型,主要用来做基数统计算法的

                                    优点: 在数据体量非常大时,计算基数所需的空间是固定的,并且很小只需要 12KB 只要花费 12KB 内存就可以计算 12^64-1 的数据量

                                    什么是计算基数: 就是计算不重复的元素 {1,2,3,3,4,5,5,8} -> {1,2,3,4,5,8}

                                    为什么用它: 如果计算 1 亿个基数,大约需要 1E/8/1024/1024 =12M 内存,如果不用 hyperLogLog 统计 10000 个对象就差不多要 120G
