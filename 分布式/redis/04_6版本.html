                # 1. Redis 6新特性

                • 用到了 IO Threads (IO多路复用)

                # 2. IO Threads (IO多路复用)

                用到这个技术的有

                • MQ: kafka,rocketMQ
                • Nginx,Tomcat
                • 技术框架: Netty

                线程介绍:

                • worker线程: 单线程的 关注请求数
                • kernel: 控制连接数
                • IO Thread: 用于搬运数据到程序中,吧数据从内核中搬运到app运行空间中

                工作流程:

                1. kernel暴露了一个接口(多路复用器 select,poll,epoll)
                2. (IO Thread) step0: worker调用内核的多路复用器来获取请求数(不包含数据的拷贝和传递)
                3. (IO Thread) step1: 读取请求数据到程序(可以是多线程也可以是单线程, redis6采用的单线程 可以配置开启)
                4. (Worker Thread) step2: 将请求放到一个线程还是多个线程,redis是放到worker线程(tomcat交给多个线程)
                5. (IO Thread) step3: 将worket线程的数据搬运到内核中(多线程),发送给网卡

                # 3. redis6.x 读请求默认关闭 IO Threads

                多读在性能上没有太大的提升:

                • 由于worker本身是单线程的,多读读到了也是要等待wk线程排队(worker线程直接读). 反而多读造成了cpu缓存成本的提升.
                • 如果在worker线程是多线程的话,那么多读的提升就很大了
                ## redis 配置文件中,搜索 io-threads 可以看到相应的描述
# 默认的线程数
io-threads 4

# 读请求默认关闭
io-threads-do-reads no

                # 4. IO多线程提高了多少性能?

                分场景, 理论值提高了 40% 在请求非对称的时候提升最大(请求的非对称性),多线程向内核写数据 提升了网卡的利用率

                为什么达不到这个理论提升?

                1. IO成本不一样. 读取请求的速度快了,但是响应的速度慢了 请求头很小,但是返回的数据很多.. 由于worker是单线程的需要串行响应数据
                  1. 打个比方请求头就2kb,但是返回的数据达到了1m get a => 'xxxx 1M的字符串'
                  2. 单线程的worker排队响应请求,需要将数据拷贝到内核空间然后通过网卡发送
                  3. 多个请求需要串行响应的过程中,是阻塞的一个状态,会造成网卡不饱满(拷贝数据1到内核->发送到网卡,拷贝数据2到内核->发送到网卡....)

                # 5. 什么场景下使用IO多线程比较好?

                发送数据量比较大的时候, 因为IO多线程主要是用于写请求(多线程向内核写数据).

                理论上任何情况下都可以开启IO多线程, 可以达到1%-40%的性能提升

                最大的提升是充分的发挥了网卡的利用率

                # 6. Tomcate, Kafka, RocketMQ的IO多线程

                one request by Thread

                NioEventLoop, eventLoop有boss和worker线程

                BossGroup:用于监听端口,如果只有一个端口 那么BossGroup开多个线程是没有意义的 WorkerGroup:用于处理事件,WorkGourp也是单线程的,用于处理socket连接,连接后将socket放到一个队列中.. 对面后面放着一个线程池用于从queue中获取socket处理IO请求

                传统的IO 一个socket就需要一个线程,Nio改变为一个线程处理多个socket连接

                # 7. Redis6的IO处理socket

                一个EventLoop监听6379, 每一个socket都是顺序处理的 Worket thread处理

                • 单线程模型: 同一个线程处理读 读完在写,然后在处理下一个socket
                • 多线程模型: 同一个线程处理读(多线程读,需要开配置),读完再写(多线程写),读或写的时候worker线程直接处理下一个socket
