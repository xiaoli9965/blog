        # 1. Redis 集群搭建(5.+)

        搭建 redis 集群环境最少需要 6 个节点完成 两台机器,6 个节点 搭建 redis 集群

        搭建redis集群前提是你已经下载了redis

        # 1.1. 创建 Redis 节点安装目录

        #创建集群文件夹
mkdir /opt/redis-cluster/
#进入集群文件夹
cd  /opt/redis-cluster/
#创建三个节点目录
mkdir 7000 7001 7002
# 拷贝解压后redis目录中src文件到redis-cluster
cp -r ../redis-5.0.8/src/ ./

        两台机器执行相应的操作

        # 1.2. 配置文件初始化

        要创建 redis 集群,必须要修改以下配置:

        官方推荐:

        • 设置端口:port 7000
        • 开启集群:cluster-enabled yes
        • cluster-config-file nodes-7000.conf
        • 节点连接超时:cluster-node-timeout 5000
        • 开启 AOF 模式:appendonly yes

        个人附加:

        • 关闭保护模式:protected-mode no
        • 后台启动:daemonize yes
        • 修改进程日志文件名与端口一致:pidfile /var/run/redis_7000.pid (不改也行)
        • 指定日志文件名:logfile "7000.log" (不写也行)
        • 指定集群环境 master 连接密码:masterauth xiaoli
        • 各各节点的连接密码:requirepass xiaoli

        🌈 将配置文件修改好之后一一存放到 7000,7001,7002 文件夹中 (配置文件在解压文件夹中)
        两台机器执行相应的操作

        # 1.3. 启动 redis 节点

        因为有两台机器,所以我们两台机器都要执行同样的操作

        #进入目录,启动节点 (两台机器执行相应的操作)
cd /opt/redis-cluster/
./src/redis-server ./7000/redis.conf
./src/redis-server ./7001/redis.conf
./src/redis-server ./7002/redis.conf

        # 1.4. 😈 创建集群(建立 Redis 主从关系)

        # 创建集群命令   -a {访问密码}
./src/redis-cli --cluster create -a xiaoli 172.16.191.128:7000 172.16.191.128:7001 172.16.191.128:7002  172.16.191.129:7000 172.16.191.129:7001 172.16.191.129:7002 --cluster-replicas 1

#根据提示输如yes就完成了集群搭建

        集群验证:

        # 通过客户端随便链接一台服务器 -c代表集群模式连接
./src/redis-cli -c -h  127.0.0.1 -p 7000 -a xiaoli
# 输入 查看当前节点信息
info replication
# 查看集群信息
cluster nodes

        • 当你连接到从节点进行写入时,会默认移动到主节点进行写入
        • 在读取时直接通过从节点进行读取 每个Redis节点都有一个永久的ID值,集群内部通过id值进行通信(而不是IP)

        集群关闭: 编写关闭脚本

        vim /opt/redis-cluster/shutdown.sh
# 输出入以下命令 ,然后保存:wq
/opt/redis-cluster/src/redis-cli -c -h  127.0.0.1 -p 7000 -a xiaoli shutdown
/opt/redis-cluster/src/redis-cli -c -h  127.0.0.1 -p 7001 -a xiaoli shutdown
/opt/redis-cluster/src/redis-cli -c -h  127.0.0.1 -p 7002 -a xiaoli shutdown
# 赋权限
chmod u+x shutdown.sh
 # 执行
./shutdown.sh
# 查看是否关闭成功
 ps -ef |grep redis
#因为有两台机器,所以两台机器执行相应操作即可

        编写启动脚本:

        # 创建文件
vim /opt/redis-cluster/start.sh
# 输如以下命令
cd /opt/redis-cluster/
./src/redis-server ./7000/redis.conf
./src/redis-server ./7001/redis.conf
./src/redis-server ./7002/redis.conf
# 赋权
chmod u+x start.sh
# 查看是否启动成功
ps -ef|grep redis
#因为有两台机器,所以两台机器执行相应操作即可

        # 1.5. 1.5.5. 集群维护

        # 1.5.1. (扩容)新增节点

        通过 --cluster add-node 新增节点时并不会分配槽,需要自己手动进行分配

        # 新增节点  add-node {要加的节点} {执行操作的节点}
 ./src/redis-cli --cluster add-node 172.16.191.129:7004 172.16.191.129:7000
# 新增slave节点  add-node {要加的节点} {执行操作的节点}
 ./src/redis-cli --cluster add-node 172.16.191.129:7004 172.16.191.129:7000 --cluster-slave

# 新增slave节点,指定主节点  add-node {要加的节点} {执行操作的节点}  --master-id {主节点ID}
 ./src/redis-cli --cluster add-node 172.16.191.129:7004 172.16.191.129:7000 --cluster-slave --cluster-master-id xxxxxxxxxxxxxxxxxxx

# 为节点分配插槽
 ./src/redis-cli --cluster reshard 172.16.191.129:7004

        # 1.5.2. (剥离)容缩

        cluster reset 清除本节点的集群信息,会退出集群

        # (容缩)剥离主节点的时候需要先干掉从节点
# --cluster del-node {执行的节点} {要删除的ID}
./src/redis-cli --cluster del-node 172.16.191.128:7002 067f4af2d27fddca7b2aa7c63fb5d4add8a47aab

        # 2. 哨兵

        redis 哨兵机制主要有三个作用:

        • 监控,通知
        • 自动做故障转移 (主挂了自动提升从为主,主活了从自动退位)
        • 配置提供

        优点: 哨兵机制是简单集群管理的利器,可以保证主从的高可用. 缺点: 只解决了高可用,并没有解决主节点的写压力(分片解决)

        在 redis 集群分片中不需要用到哨兵,因为那个里面的高可用机制类似哨兵并且更强大

        # 3. 集群分片

        • 集群会自动进行故障转移,主挂了从会顶上. 但是在这个过程中会丢失部分请求(无法避免),当主在次起来后从自动退位
        • 集群中会自动平衡主从关系(默认每个主都会分配一个从). 比如:A 主有 2 个从,B 主只有一个从 当 B 主挂了后 B 从升级为主,次数集群会将 A 主其中的一个从一到 B 来当从
        • 在代码中连接集群时碰到节点挂了的情况连接会自动切换到可用节点中
        • 分片也是有弊端的, 比如要做一个取交/差集: 如果一个 KEY 在分片 1 一个 KEY 在分片 2 这个时候是不支持做的,需要用到 TAG 标签使 key 都落在一个分片中 例: USER:INFO:I 中的 USER:INFO就是 TAG

        # 3.1. 投票机制

        1. 当一个节点通信超时cluster-node-timeout后,会有一个 master 节点发起广播投票
        2. 超过半数及诶单认同这个节点挂调之后即整个集群认定这个机器挂了(节点下线,从节点顶上)
        3. 当整个集群半数以上节点挂掉后整个集群不可用

        # 4. 短信验证码

        准备两个 Key: 过期时间 2 分钟

        • 发送次数 KEY:VERIFY:PASSWORD_MODIFY:132XXXXX:COUNT

        • 验证码 KEY:VERIFY:PASSWORD_MODIFY:132XXXXX:CODE

          1.用户请求发送验证码时先查 Redis 判断发送次数有没有到达限制,到达了次数限制就拒绝 2.没有到达限制就生成验证码保存到 Redis 3.判断用户输入的值,如果匹配就 删除发送次数KEY和验证码KEY
