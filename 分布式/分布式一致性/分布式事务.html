                # CAP 定理

                • 一致性 C (Consistency)
                  • 强一致性
                  • 弱一致性(最终一致性)
                • 可用性 A (Availability)
                • 分区容错性 P (Partition tolerance)

                要达到良好的响应性能来提高用户体验，因此一般都会选择AP保证最终一致性。

                # 强一致性

                • Paxos 算法
                • 主从同步
                • Raft 算法
                • ZAB

                # 弱一致性(最终一致性)

                • DNS
                • Gossip

                # BASE 理论

                • 基本可用 (Basically Available)
                • 软状态 (Soft state)
                • 最终一致性 (Eventual consistency)

                Base理论是对CAP中AP的一个扩展，通过牺牲强一致性来获得可用性，当出现故障时要保证核心功能可用，允许数据在一段时间内的不一致，但最终达到一致性的状态，也叫做柔性事务

                # 方式一: XA 规范/JTA 协议

                需要数据厂商支持 XA 规范,需要 Java 组件有 atomikos 等

                基于数据库的XA协议来实现的2PC方案就叫XA

                具体交互

                1.TM（事务管理器）向AP（应用）提供编程接口，AP通过TM提交及回滚事务
                2.TM通过XA协议来通知数据库进行事务的开启、关闭、提交、回滚

                流程

                1.准备阶段：RM会执行实际的业务操作，但不会提交事务（会锁定资源）
                2.提交阶段：RM接收到准备阶段执行回复，然后进行事务提交或回滚操作（释放资源）

                # 方式二: 异步校对数据

                在支付行业比较多,对账单形式

                # 方式三: 可靠消息(MQ) 💥

                异步场景;通用性强;扩展性高;

                # 方式四: TCC 编程式

                一般来讲需要有一套框架支撑..需要公司自己定义协议

                两阶段提交补偿性主要分为两步实现

                try阶段、confirm阶段/Cancel阶段。

                • try阶段：主要对系统资源进行预留和检查
                • confirm阶段：当try阶段全部成功时就会执行confirm阶段事务提交。默认情况下只要try成功了就默认confirm阶段一定会成功
                • concle阶段：当try阶段有失败的时候就会执行concel阶段进行回滚，释放预留资源

                方案成熟

                # 设计思路

                # 步骤 1-可靠消息生产-记录消息发送

                • 把要发送的数据记录到数据库(消息记录表)
                • 通过定时任务不断轮询判断消息是否发送成功,并记录发送次数
                • 没法成功就重发,并记录次数. 要保证一定要吧数据发过去,重试次数太多就出发报警通知运维
                • 通过发布者确认机制收到ACK后即代表消息成功发送

                # 步骤 2-修改发送状态

                利用 RabbitMQ 的消息确认机制

                如果消息成功发送,收到了 ACK 之后将消息发送状态改为已发送

                # 步骤 3-可靠消息处理

                • 开启手动 ACK. 由消费者控制消息的重发/清除/丢弃.
                • 幂等性问题. 防止重复处理,根据业务 ID 或业务数据判断消息是否重复,重复就不处理

                处理过程:

                • 当消息正常处理后,就手动确认ACK
                • 如果消息处理失败,返回NACK请求消息重发
                  • 如果多次尝试一直失败,就进行告警人工干预 或者 放弃操作(需要保留数据)

                # 本地消息表方式

                • 通过不断的扫描消息表并发现送消息到MQ中，监听MQ的应用通过ACK机制保证消息一定能够消费到;
                • RocketMQ事务消息方案（需要做幂等判断）：1.事务发起时向MQ中发送消息，2.当本地事务执行完毕后返回Commit将MQ中的消息改为可消费状态， 3.事务参与者（其他监听MQ的服务）获取到消息后进行相应的操作
